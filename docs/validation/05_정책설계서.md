## C-1 (상) 닉네임 길이 규칙 단일화

### 쟁점 / 현황
같은 "닉네임 길이" 정책이 영역·화면별로 3중 불일치한다.
- AD01 P-AD-10: 화면 안내 "2~20자" vs Description "2~10자" (한 화면 안에서도 충돌, `[미확정]`).
- AD01 P-AD-09 회원가입 검증: 닉네임 "2~10자, 금칙어".
- FR01 P-46 / P-AD 프로필(p150 카운트 [n/15]) / FR01 닉네임 변경 모달(S-FR01-0301-112-P02): "2~15자".

### 결정 (권고안)
**전 영역(Front·Admin·Brand 공통) 닉네임을 `2자 이상 15자 이하`로 단일 규칙 확정.** 최소 2자, 최대 15자, 공백 단독·금칙어 불가(기존 정책 유지).

### 근거
| 후보 | 장점 | 단점 | 채택 |
|------|------|------|:--:|
| 2~10 | 짧고 목록 정렬 안정 | Front 프로필/변경모달(2~15) 이미 구현 가정과 충돌, 한글 닉네임에 빡빡 | — |
| **2~15** | Front 다수 화면(P-46·변경모달·프로필 카운트 [n/15])이 이미 채택, 변경 최소 | Admin 가입검증 문구만 수정하면 됨 | **○** |
| 2~20 | 여유 | 어떤 화면도 일관 미사용, DB·목록 UI 폭 부담 | — |

- 일관 우선: 화면 다수가 이미 사용 중인 값(2~15)으로 수렴하는 것이 수정 범위가 가장 작다(Admin 문구 2곳만 교체).
- 측정가능·명문화: 경계값 = 1자(차단)/2자(허용)/15자(허용)/16자(차단). QA 경계값 테스트 가능.

### 영향 화면·문서

## M-2 (상) 강사/서브강사 LNB 메뉴 권한(RBAC) 기준 설계

### 쟁점 / 현황
- AD01 §0-2·P-AD-02: 어드민 접속 주체는 관리자/강사(서브강사 포함)로 추정되나 **강사/서브강사가 보는 LNB 메뉴·기능 범위 미정의**(`[미확정]`).
- 단, 권한 부여 메커니즘은 이미 존재: P-AD-19 권한 설정 모달 = **메뉴 권한 체크트리(상위→하위 연동), [설정 완료] 시 이전 권한 무시·일괄 갱신**. 강사/관리자 기본정보 탭에 "접근 가능 메뉴 목록 + [권한 설정]".
- LNB 1depth 11개: 대시보드 · 사용자 · 상품 · 콘텐츠 · 판매 · 운영 · 마케팅 지원 · 사이트 디자인 · 통계 · 매출 및 정산 · 설정.

### 결정 (권고안)
**역할 기반 "메뉴 화이트리스트 RBAC + 관리자 개별 가감"** 모델로 확정. 즉 역할별 **기본 권한 세트(default)** 를 정의하고, P-AD-19 체크트리로 계정별 가감(관리자가 강사에게 메뉴를 추가/회수). LNB는 **권한 없는 메뉴를 숨김(hide)** 한다(비활성 노출 아님 — 정보 누수 방지).

역할별 기본 권한 세트(default, 관리자가 P-AD-19로 조정 가능):

| LNB 1depth | 관리자(소유자) | 강사(default) | 서브강사(default) |
|------------|:--:|:--:|:--:|
| 대시보드 | ● | ●(본인 강좌 범위) | ●(본인 강좌 범위) |
| 사용자(학습자/강사/관리자) | ● | △ 학습자(본인 강좌 수강생)만 | ✕ |
| 상품 | ● | ●(본인 담당 상품) | ●(본인 담당 상품·등록/수정) |
| 콘텐츠(라이브러리) | ● | ● | ● |
| 판매(주문/쿠폰/환불) | ● | △ 본인 상품 주문 조회 | ✕ |
| 운영(게시판/팝업) | ● | △ 본인 커뮤니티/게시판 | △ |
| 마케팅 지원 | ● | ✕(기본) | ✕ |
| 사이트 디자인 | ● | ✕ | ✕ |
| 통계 | ● | △ 본인 강좌 통계 | ✕ |
| 매출 및 정산 | ● | △ **본인 정산 정보·내역만** | ✕ |
| 설정 | ● | ✕(공지 확인만) | ✕ |

범례: ● 전체 / △ 본인 범위 한정(데이터 스코프 필터) / ✕ 비노출.

핵심 규칙:
1. **관리자(소유자)=전체 권한**. 강사/서브강사는 위 default에서 시작, 계정별 가감(P-AD-19).
2. **데이터 스코프**: 강사가 권한을 가진 메뉴라도 데이터는 "본인 담당 상품/강좌/수강생/정산"으로 한정(전사 데이터 접근 금지). 메뉴 권한(보이는가)과 데이터 스코프(무엇이 보이는가)를 분리.
3. **서브강사 = 강사 − 운영성/정산/사용자 관리**(콘텐츠·상품 작업 중심 보조 역할).
4. **민감 메뉴 기본 차단**: 매출/정산(전사)·사이트 디자인·마케팅 발송(크레딧 차감)·관리자 관리는 강사 default에서 제외(관리자가 명시 부여해야 노출).
5. **권한 없는 진입 차단**: 직접 URL 접근 시 EXC-06(접근 차단) 적용.

## R-1 (발산·선합의 채택) 알림 단일 도메인 — 라우팅 테이블 통합

### 쟁점 / 현황 (현재 구조)
알림이 영역마다 따로 산다 — Admin 알림센터(SUP-07, M-4 트리거 카탈로그)·Front 알림센터(SUP-07)·학습자 알림 채널 설정(SUP-09)·Front 자동메일(M-5)·강사/관리자 공지(SUP-08)·푸시(DeviceToken). 같은 **"이벤트 → 수신자 → 채널 전달"**인데 여러 화면·여러 정책으로 흩어져 각각 트리거/채널/수신거부 규칙을 따로 정의하면, 정보통신망법 컴플라이언스 누락·중복 정의·추적성 약화 위험이 크다.

## C-6 (하) 본인 화면 마스킹 예외 범위

> 정책ID: CMP-01 · CMP-02 · OQ-CMP(06) · 표준 §7 | 분류: ⓐ DJ 추인(컴플라이언스, biz-legal 확인 권고)

### 쟁점 / 현황
- 표준 §7은 **항상 마스킹**. 그러나 프로필/1:1문의/수료증의 본인 정보·계정관리 본인 이메일은 "**본인이 자기 정보를 보는 화면**"이라 비마스킹 가능성(CMP-02·04 C-6). 표준과 충돌 가능.

### 결정 (운영자 컨펌 완료 · 2026-06-25 — 보수안)
- **본인이 자기 정보를 보는 화면은 비마스킹 허용** — 본인 이름/이메일/휴대폰/닉네임은 전체 표기(자기 정보 확인 UX).
- **단, 화면 캡처·공유·어깨너머 유출 위험이 큰 금융 정보(카드번호·계좌번호)는 본인 화면에서도 부분마스킹 유지** — 카드번호 뒤 4자리(`**** **** **** 1234` CMP-01)·계좌번호 부분마스킹(표준 §7).
- **비밀번호·인증코드 입출력 마스킹은 본인 화면에서도 예외 없이 항상 유지.**

### 근거
| 후보 | 장점 | 단점 | 채택 |
|------|------|------|:--:|
| 전 항목 항상 마스킹(표준 그대로) | 일관·안전 | 본인이 자기 이메일·이름도 못 봄 → UX 비합리 | — |
| 본인 화면 전면 비마스킹 | UX 자연스러움 | 화면 캡처/공유·어깨너머 시 카드/계좌 노출 위험 | — |
| **본인 화면 비마스킹 허용 + 카드/계좌번호는 부분마스킹 유지(보수안)** | UX·보안 균형·캡처 유출 위험 최소화 | 항목별 분기 규칙 필요 | **○** |

- 표준 §7의 취지는 "타인·제3자 노출 방지"인데 본인 화면의 본인 식별정보는 보호 대상이 본인 → 비마스킹이 합리.
- 카드/계좌는 본인 화면이라도 캡처 공유·화면 녹화 시 금전 피해로 직결 → 보수적으로 부분마스킹 유지(정보통신망법·개인정보처리방침과 정합, CMP-01, 표준 §7 정합).
