## C-1 (상) 닉네임 길이 규칙 단일화

### 쟁점 / 현황
같은 "닉네임 길이" 정책이 영역·화면별로 3중 불일치한다.
- AD01 P-AD-10: 화면 안내 "2~20자" vs Description "2~10자" (한 화면 안에서도 충돌, `[미확정]`).
- AD01 P-AD-09 회원가입 검증: 닉네임 "2~10자, 금칙어".
- FR01 P-46 / P-AD 프로필(p150 카운트 [n/15]) / FR01 닉네임 변경 모달(S-FR01-0301-112-P02): "2~15자".

### 결정 (권고안)
**전 영역(Front·Admin·Brand 공통) 닉네임을 `2자 이상 15자 이하`로 단일 규칙 확정.** 최소 2자, 최대 15자, 공백 단독·금칙어 불가(기존 정책 유지).

### 근거
| 후보 | 장점 | 단점 | 채택 |
|------|------|------|:--:|
| 2~10 | 짧고 목록 정렬 안정 | Front 프로필/변경모달(2~15) 이미 구현 가정과 충돌, 한글 닉네임에 빡빡 | — |
| **2~15** | Front 다수 화면(P-46·변경모달·프로필 카운트 [n/15])이 이미 채택, 변경 최소 | Admin 가입검증 문구만 수정하면 됨 | **○** |
| 2~20 | 여유 | 어떤 화면도 일관 미사용, DB·목록 UI 폭 부담 | — |

- 일관 우선: 화면 다수가 이미 사용 중인 값(2~15)으로 수렴하는 것이 수정 범위가 가장 작다(Admin 문구 2곳만 교체).
- 측정가능·명문화: 경계값 = 1자(차단)/2자(허용)/15자(허용)/16자(차단). QA 경계값 테스트 가능.

### 영향 화면·문서
- 수정: AD01 P-AD-09 회원가입 검증(2~10→2~15), AD01 P-AD-10 화면 안내·Description(2~20/2~10→2~15), S-AD01-0303-001, S-AD01-9003-001.
- 정합 확인(유지): FR01 P-46 [S-FR01-0301-112], 닉네임 변경 모달 [S-FR01-0301-112-P02].
- 04 정책요약 AUTH-15 비고 갱신.

### 후속 위임
`[→ 송기획: AD01 P-AD-09/10 닉네임 길이 문구·카운트를 2~15로 통일 반영]` `[→ 강테크: 06_API계약 닉네임 validation min=2,max=15 명시]`

### 철회 조건

## M-2 (상) 강사/서브강사 LNB 메뉴 권한(RBAC) 기준 설계

### 쟁점 / 현황
- AD01 §0-2·P-AD-02: 어드민 접속 주체는 관리자/강사(서브강사 포함)로 추정되나 **강사/서브강사가 보는 LNB 메뉴·기능 범위 미정의**(`[미확정]`).
- 단, 권한 부여 메커니즘은 이미 존재: P-AD-19 권한 설정 모달 = **메뉴 권한 체크트리(상위→하위 연동), [설정 완료] 시 이전 권한 무시·일괄 갱신**. 강사/관리자 기본정보 탭에 "접근 가능 메뉴 목록 + [권한 설정]".
- LNB 1depth 11개: 대시보드 · 사용자 · 상품 · 콘텐츠 · 판매 · 운영 · 마케팅 지원 · 사이트 디자인 · 통계 · 매출 및 정산 · 설정.

### 결정 (권고안)
**역할 기반 "메뉴 화이트리스트 RBAC + 관리자 개별 가감"** 모델로 확정. 즉 역할별 **기본 권한 세트(default)** 를 정의하고, P-AD-19 체크트리로 계정별 가감(관리자가 강사에게 메뉴를 추가/회수). LNB는 **권한 없는 메뉴를 숨김(hide)** 한다(비활성 노출 아님 — 정보 누수 방지).

역할별 기본 권한 세트(default, 관리자가 P-AD-19로 조정 가능):

| LNB 1depth | 관리자(소유자) | 강사(default) | 서브강사(default) |
|------------|:--:|:--:|:--:|
| 대시보드 | ● | ●(본인 강좌 범위) | ●(본인 강좌 범위) |
| 사용자(학습자/강사/관리자) | ● | △ 학습자(본인 강좌 수강생)만 | ✕ |
| 상품 | ● | ●(본인 담당 상품) | ●(본인 담당 상품·등록/수정) |
| 콘텐츠(라이브러리) | ● | ● | ● |
| 판매(주문/쿠폰/환불) | ● | △ 본인 상품 주문 조회 | ✕ |
| 운영(게시판/팝업) | ● | △ 본인 커뮤니티/게시판 | △ |
| 마케팅 지원 | ● | ✕(기본) | ✕ |
| 사이트 디자인 | ● | ✕ | ✕ |
| 통계 | ● | △ 본인 강좌 통계 | ✕ |
| 매출 및 정산 | ● | △ **본인 정산 정보·내역만** | ✕ |
| 설정 | ● | ✕(공지 확인만) | ✕ |

범례: ● 전체 / △ 본인 범위 한정(데이터 스코프 필터) / ✕ 비노출.

핵심 규칙:
1. **관리자(소유자)=전체 권한**. 강사/서브강사는 위 default에서 시작, 계정별 가감(P-AD-19).
2. **데이터 스코프**: 강사가 권한을 가진 메뉴라도 데이터는 "본인 담당 상품/강좌/수강생/정산"으로 한정(전사 데이터 접근 금지). 메뉴 권한(보이는가)과 데이터 스코프(무엇이 보이는가)를 분리.
3. **서브강사 = 강사 − 운영성/정산/사용자 관리**(콘텐츠·상품 작업 중심 보조 역할).
4. **민감 메뉴 기본 차단**: 매출/정산(전사)·사이트 디자인·마케팅 발송(크레딧 차감)·관리자 관리는 강사 default에서 제외(관리자가 명시 부여해야 노출).
5. **권한 없는 진입 차단**: 직접 URL 접근 시 EXC-06(접근 차단) 적용.

### 근거
- 보수적·최소권한 원칙: 정산/매출/마케팅 발송은 금전·대외 영향이 커 기본 차단 후 명시 부여가 안전.
- 기존 메커니즘 재사용: P-AD-19 체크트리가 이미 "상위→하위 연동·일괄 갱신"을 제공 → default 세트만 정의하면 화면 추가 없이 정책 완결.
- 강사 상세 [정산 정보] 탭(p113)은 "본인 정산"이므로 강사 default에 본인 범위 정산을 허용하는 것이 일관.
- 측정가능: 역할별 노출 메뉴 = 위 표로 RBAC 테스트케이스화(403 인가 검증, QA 표준 §7.2 ②).

### 영향 화면·문서
- AD01 P-AD-02(LNB 권한별 노출), P-AD-19 [S-AD01-0102/0103](권한 모달 default 프리셋), 강사/관리자 기본정보 탭 "접근 가능 메뉴 목록".
- AD01 대시보드 진입(P-AD-04) "강사 계정 진입 동작 `[미확정-강사권한]`" 해소.
- 04 정책요약 AUTH-11·AUTH-12 비고 갱신.

### 후속 위임
`[→ 운영자: 역할별 default 권한 세트(특히 강사 정산/통계 노출 범위) 컨펌]` `[→ 송기획: P-AD-02 LNB를 역할별 노출로 화면 명세, P-AD-19에 default 프리셋 반영]` `[→ 강테크/오백개: 06_API계약에 RBAC(메뉴 권한 + 데이터 스코프) 미들웨어·역할별 403 규칙 명시]`

### 철회 조건
운영 결과 강사에게 더 넓은 자율(예: 본인 마케팅 발송)이 필요하다고 판단되면 default 세트를 확장하되, 정산/사용자 관리/사이트 디자인의 전사 범위는 관리자 전용 유지를 권고.

### 운영자 컨펌
**완료(2026-06-25) — 임기획 default 안 그대로 확정.** 강사=본인 범위(대시보드·상품·콘텐츠·본인 정산·본인 강좌 통계·주문 조회) 허용 / 마케팅·사이트디자인·사용자관리·전사 정산 차단. 관리자가 P-AD-19로 계정별 가감.

## C-4 (중) 쿠폰 할인 타입 오픈 범위

### 쟁점 / 현황
- AD01 P-AD-66(MKT-14): 쿠폰 발행 **"할인 타입 정액 only(정률 추후 확장)"**.
- FR01 P-41(MKT-15): 쿠폰 카드 UI에 **"정률 {n}% / 정액 -{n}원"** 양쪽 표기.
- 즉 Admin은 정액만 발행 가능한데 Front 카드 UI는 정률도 표시 → 발행 불가능한 타입을 노출하는 불일치.

### 결정 (권고안)
**오픈 시점은 정액(정액 할인) only로 확정.** Front 쿠폰 카드의 **정률(%) 표기는 숨김(hide)** 처리하고, 정률 표기/렌더 로직은 코드에 남기되 비활성. **정률 할인은 P1으로 이관**(Admin 발행 + Front 표기 동시 오픈).

### 근거
| 후보 | 장점 | 단점 | 채택 |
|------|------|------|:--:|
| 정액·정률 동시 오픈 | 마케팅 유연성↑ | Admin 발행 UI·검증·정산(환불 일할 시 정률 계산) 추가 필요, 오픈 일정 리스크 | — |
| **정액 only(정률 P1)** | Admin 현 구현과 일치, 오픈 범위 최소·안정 | 정률 프로모션 일시 불가 | **○** |

- 일관·보수 우선: 백오피스(발행)가 정액만 지원하므로 Front도 정액만 노출하는 것이 "발행 불가 타입 노출" 모순 제거.
- 정률은 환불 일할·최소주문금액·중복할인 상호작용 등 부수 정책이 더 필요 → 별도 라운드(P1)가 안전.

### 영향 화면·문서
- MKT-14·MKT-15. FR01 P-41 [S-FR01-0301-105](정률 표기 숨김), AD01 P-AD-66 [S-AD01-0402-001/002](정액 only 유지·정률 비고 P1).
- PAY-18(결제 시 쿠폰 1개 적용)은 영향 없음.
- 04 정책요약 MKT-14·MKT-15 비고 갱신.

`[→ 송기획: FR01 쿠폰 카드 정률(%) 표기 분기 숨김, Admin 쿠폰 발행 정액 only 명시·정률 P1 주석]` `[→ 강테크: 06_API계약 쿠폰 discountType=AMOUNT only(향후 RATE 확장 자리 확보)]`
