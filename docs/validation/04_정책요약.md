> 이 문서는 파이프라인 **단계 ③(정책 합본)**의 산출물이다. 01~03 화면설계서에 흩어진 정책 규칙을 7개 도메인으로 모아 **중복 제거·충돌 정렬**한다.
> 각 정책 항목에 **출처 화면ID(추적키)**를 단다. 같은 정책이 여러 영역에 나오면 하나로 통합하고 **영역 교차 표기**(`[FR01-…, AD01-…]`)한다.
> 결제/구독 정책은 `billing.md`·회사정보 표준과 정합 확인했으며, 불일치는 §6·§8에 표로 명시한다.

---

## 작성 규칙

- 한 행 = 검증 가능한 정책 1건. 골격: `정책ID | 정책 내용 | 출처(영역 교차) | 비고`.
- 출처는 원문 정책ID(P-xx / P-AD-xx / P-AUTH-xx)와 화면ID를 함께 적어 추적성 유지.
- 충돌(영역 간 상이)·미결은 본문에서 빼서 §6(충돌/미결)·§7(발산형)에 모은다.
- 결제·구독은 `billing.md`(1차 SoT) > 회사정보-맑은소프트.md §3 순으로 정합 기준.

## 9. 발산형(Rethink) 제언 종합

> 3영역에서 도출된 구조적 제언을 모은다. 각 항목: 현재 구조 → 제안 구조 → 왜 더 나은가 → 비용/리스크. '상/중/하' 심각도와 별개로 항상 노출(QA 표준 §8.3).

### 9.1 [BR01] Brand 마이페이지군 → 계정/구독 콘솔 분리
- **현재**: 결제내역·문의내역·이용상품·계정관리·내 사이트 관리가 brand(마케팅) 사이트 GNB(로그인 후) 하위 BR01로 일괄 수용.
- **제안**: 별도 업무구분(예 `MY01` 또는 콘솔 영역)으로 분리 → IA가 "마케팅 LP(BR01) ↔ 계정/구독 콘솔(MY01)"로 명확.
- **왜 더 나은가**: 성격(마케팅 vs 계정/결제 운영)이 본질적으로 다름. 화면ID 추적·소유권 경계가 깨끗해지고 §8 M-10 해소.
- **비용/리스크**: 화면ID 재부여(약 10화면)·교차 링크 재정의. → 팀장 컨펌(§0-2 연동).

### 9.2 [BR01] 약관 5종 페이지 → 단일 약관 페이지 + 탭/앵커 통합
- **현재**: 이용약관·무료약정·유료약정·개인정보처리방침·마케팅 수신동의 5개 풀페이지(S-BR01-1101-001~005).
- **제안**: 단일 약관 페이지 + 탭/앵커. 본문은 JSON SoT이므로 탭 전환만으로 충분.
- **왜 더 나은가**: 화면 5개→1개, 중복 레이아웃 spec 제거, 개정일 드롭다운 1곳 관리.
- **비용/리스크**: 직접 URL(SEO·푸터 링크) 호환을 위해 앵커/딥링크 유지 필요. 낮음.

### 9.3 [BR01] 미설계 LP 3종(프로덕트/데모보기/주요기능) 흡수
- **현재**: 프로덕트=빈 GNB 프레임(미설계), 데모보기·주요기능=실사이트 대체.
- **제안**: main/플랫폼소개로 흡수하거나 라이브 링크로 대체해 화면 수 축소.
- **왜 더 나은가**: 유령 화면(설계 없는 메뉴) 제거로 IA 정직성↑·관리 비용↓.
- **비용/리스크**: GNB 메뉴 재배치. 마케팅 콘텐츠 전략 결정 필요(§0-6). [→ 임기획/마케팅]

### 9.4 [전사] 단방향 피드백 패턴 토스트 일원화 (Front→전 영역)
- **현재**: Front만 얼럿 폐지·토스트 일원화(P-02). Brand/Admin은 Alert(MPU) 유지(C-5).
- **제안**: Brand/Admin의 단방향 고지도 토스트로 통일(양방향만 컨펌 유지).
- **왜 더 나은가**: 3영역 인터랙션 일관성, 사용자 학습비용↓. 공통 컴포넌트 단일화.
- **비용/리스크**: 기존 Brand/Admin Alert 호출부 전수 교체. 중. [→ 윤UX/임기획]

### 9.5 [전사] 정책 ID 네이밍 통일 (추적성 강화)
- **현재**: 영역별 정책ID가 P-01(Front)·P-AD-01(Admin)·P-AUTH-01(Brand)로 접두사·체계 상이.
- **제안**: 본 04 문서의 도메인 접두사(AUTH/PAY/LRN/MKT/CMP/EXC/SUP)를 전사 표준 정책ID로 승격, 영역 원문ID는 출처(추적키)로만 유지.
- **왜 더 나은가**: 06_API계약·QA 테스트케이스에서 도메인 단위로 정책을 1:1 추적(누락 0). FR-ID 추적성 매트릭스(DOCS 표준 §3.1) 정합.
- **비용/리스크**: 본 문서가 이미 그 골격 — 후속 문서가 04 정책ID를 인용하도록 강제만 하면 됨. 낮음.
