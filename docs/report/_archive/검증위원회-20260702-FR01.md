## 2. 보안 위원 — 조건부 GO

- **견고(⭕)**: CORS 화이트리스트(`solsol.pages.dev` 허용·`attacker.test` 거부·와일드카드+credentials 없음) · JWT(서명·exp·교차테넌트 tid/sch·refresh 회전+재사용탐지+pepper) · 인가(보호 라우트 401·roleGuard·/me sub 기반 IDOR 방어) · 시크릿 무노출·`/ops` 프로덕션 403.
- **운영 전 필수 보완**:
  - **SEC-A (상·구조·오너)** — refresh 쿠키가 `pages.dev↔workers.dev` 서드파티 쿠키(SameSite=None). Safari ITP/Chrome 차단 시 세션 갱신 단절. 동일 상위도메인 서브도메인 배치 or 수용 여부 오너 결정.
  - **SEC-C (중)** — `/health/db`·`/health/db/grants`가 무인증 공개 → DB 스키마명·MySQL 버전·**GRANT 전문** 노출. 프로덕션 차단/게이트 필요.
- **후속(블로커 아님)**: SEC-E(login·reset 레이트리밋·AUTH-07 발송제한 미구현) · SEC-D(프론트 access localStorage → XSS 15분 토큰, refresh는 HttpOnly 보호) · SEC-B(크로스도메인 유지 시 refresh CSRF 토큰 부재) · dba 앱 DB 계정 최소권한화.
- 상세: `docs/dev-validation/committee-fr01-security.md`

## 5. 결론

FR01 사용자단은 **화면(48)·정책(05 6건)·개인정보 게이트를 blocker 0으로 통과**했고, 실 API 결선·배포 상태에서 **핵심 보안 통제가 견고(critical 0)**하다. 따라서 **데모/프리뷰 운영 GO**. **실서비스 오픈**은 §4 「운영 전 필수」 3건(쿠키 도메인·health 차단·동의 이력) 해소 및 오너 결정(PG·R2·실 테넌트) 완료를 게이트로 한다.

> 한계 고지: 본 검증은 정본 대조·코드패턴·typecheck·라이브 프로브 기반이며, **런타임 브라우저 전수 스모크(상태축 실측)** 는 실연동 라운드에서 별도 수행을 권고한다.
