# 개발–검증 연동 절차 (Dev × Validation) — 단일 정본

쏠쏠 크리에이터/브랜드 제품의 모든 개발은 **[docs/validation/](validation/)** 검증 패키지를
**정본(SoT)** 으로 삼아 "제대로 개발되고 있는지"를 추적키(화면ID·엔드포인트) 단위로 확인한다.

> **이 문서가 검증 절차의 유일 정본이다.** CLAUDE.md의 "개발–검증 연동"·"목업/구현 검증 절차(라운드)"
> 절은 요약·링크이며, 상세·판정 기준은 항상 본 문서를 따른다.
> (2026-07-01 오너 확정 — [메모리: validation-process-decisions] 참조. 팀 논의: qa·security·privacy·planner·dba.)

## 0. 원칙 (불변)

- **검증 폴더는 읽기 전용 정본**: `docs/validation/` 의 어떤 파일도 개발 중 **수정·생성·삭제하지 않는다.**
  변경이 필요하면 **기획-lead/총괄(오너) 승인 후 별도 절차**로만 처리한다.
- **추적키 = 라운드의 뼈대**: 화면 작업은 **화면ID**(`S-FR01-…`·`S-AD01-…`·`S-BR01-…`), 백엔드/계약 작업은
  **엔드포인트ID(+영향 화면ID)**, 스키마 작업은 **테이블/마이그레이션**을 명시한다. SoT ↔ 구현 ↔ 결함표가 동일 키로 매핑(누락 0)돼야 한다.
- **권위(authority) 분리** — 충돌 시 우선순위:
  1. **`05_정책설계서.md` 확정 6건** (충돌 최종값, 최우선)
  2. 구조·상태·IA·라우팅 = **화면설계서 `00~03`**
  3. 동작·규칙 = **`04_정책요약.md`** (+ 05)
  4. 정확한 문구·최종 비주얼 = **참조 캡처 PNG**(`_exports/png/…`)
  5. 토큰값(색/간격/타이포) = 스타일가이드
  - **데이터(물리) 권위**: 스키마·타입·제약·인덱스의 정본은 **구현 DDL**(`master.sql`·`tenant_template.sql`) → **`ERD.md`**(파생 뷰)다.
    **`06_API계약.md`는 D1 가정으로 작성돼 데이터 정본이 아니다**(2026-07-01 격하). 06은 엔드포인트 계약 **참고**로만 쓴다. → 백엔드는 Aurora MySQL schema-per-tenant([메모리: solsol-backend-aurora-not-d1]).
- **`[추정]`(OCR 본문)은 권위가 아니다** — 문구·정책을 단정하지 말고 PNG 원본/05로 확정한다.
- **비가역 행위 게이트**(계정삭제·즉시환불·실결제 최종승인·데이터 영구삭제)는 절차와 무관하게 **항상 사용자/운영자 확인** 전제 — 자동 진행 금지.

### 05 확정 6건 (개발 판정 최종 기준 — 위반 시 결함)
1. **닉네임** 전 영역 **2~15자** 단일 규칙
2. **무료체험 미운영** (가입=Free 즉시부여 / 유료=즉시결제, 14일 체험 UI 없음이 정상)
3. **강사/서브강사 = 메뉴 화이트리스트 RBAC** (관리자=전체 / 강사=본인범위 / 서브강사=강사−운영·정산·사용자관리) — **데이터 스코프(own) 서버 강제 포함**
4. **유효시간**: 가입코드 10분 / 비번재설정 30분 / 초대 48시간

6. **쿠폰 정액 only** (Front 정률 표기 숨김)

### 검증 제외(있으면 오히려 결함/대상 아님)

## 1. 검증 적용 대상 매트릭스 (레포 × 트랙)

검증은 **두 트랙**으로 나뉜다 — 화면(프론트) 라운드와 계약(백엔드/API) 라운드. 추적키·정본·게이트 주체가 다르다.

| 레포 | 트랙 | 추적키 | 정본(권위) | 비고 |
|---|---|---|---|---|
| `solsol` (FR01) | 화면 라운드 | 화면ID `S-FR01` | 00~03·04·05·PNG | 크리에이터 사용자단 |
| `solsol-admin` (AD01) | 화면 라운드 | 화면ID `S-AD01` | 상동 | 크리에이터 관리자단 |
| `solsol-brand` (BR01) | 화면 라운드 | 화면ID `S-BR01` | 상동(+제외목록) | 브랜드 사용자단(순수 프론트) |
| `solsol-brand-admin` (BA01) | 화면 라운드 (+실연동 시 §2 stage8) | 화면ID `S-BA01` | 상동 | 풀스택 운영자단 · 인증/세션/RBAC/PII는 실연동 대상, 데이터 정본은 brand-api(master DDL)에 귀속 |
| `solsol-api` | 계약 라운드 | 엔드포인트ID(+영향 화면ID) | **ERD.md·실 API 계약** (06은 참고) | + 데이터 5축 |
| `solsol-brand-api` | 계약 라운드 | 엔드포인트ID(+영향 화면ID) | **ERD.md·실 API 계약** (master DDL) | + 데이터 5축 |
| (스키마·data-model) | 데이터 라운드 | 테이블·마이그레이션 | **DDL·ERD.md** | dba 주체, 데이터 5축 |
| `solsol-mng` (관리 허브) | **라운드 비적용** | — | — | 자체 [배포 일괄 절차] 스모크만 |

> 통과 스냅샷에 **테이블 수 + 커밋 해시**를 기록해 "무엇을 통과했는지" 고정한다(스키마 표류 방지).

## 2. 단계별 검증 프로파일 (WBS 8단계 정합)

같은 "라운드"라도 산출물 단계에 따라 검증 강도·주체·산출물이 다르다. 획일 적용하지 않는다.

| WBS 단계 | 검증 프로파일 | 축 | 게이트 주체 | 종료조건 |
|---|---|---|---|---|
| **4 목업/퍼블** | 목업 라운드 | 9축 중 존재·IA/라우팅·카피·토큰·반응형 중심 + 05 6건 | qa (단독 가능) | ❌'상' 0건 · **오너 승인 시 미통과 배포 허용**(시연용) |
| **5 설계** | 설계 리뷰 | 구조·경계·정합(발산형 제언) | architect·dev-lead | 경량(합의) |
| **6 스파이크(1차 개발)** | PoC 검증 | 가설 검증·리스크 축소 위주(경량) | 담당 개발자 + qa 스팟 | 경량(가설 판정) |
| **7 구현** | 구현 라운드 | **화면 9축 전체** + 05 6건 | qa · security · privacy (3렌즈) | **❌'상' 0건 강제** |
| **7 구현(백엔드/API)** | 계약 라운드 | **데이터 5축** + 계약정합(엔드포인트·요청/응답·에러코드) + 마스킹 | dba · security · qa | **❌'상' 0건 강제** |
| **8 운영·계약/실연동** | 실연동 라운드 | 실 PII·시크릿·CORS·테넌트 격리·PG·레이트리밋 + 실측 마스킹 | **security · privacy 서명 필수** + qa | **❌'상' 0건 강제**(우회 불가) |

- **미통과 배포 우회는 목업/시연 라운드만**, 그것도 **오너 승인 기록** 시에만. **실앱·실연동은 blocker 0을 강제**한다(우회 불가).
- **실연동 라운드**(인증·세션·인가·시크릿·PG·테넌트·PII를 건드리는 작업)는 **security·privacy 서명 없이는 GO 불가**. QA 단독 서명 금지.

## 3. 검증축

### 3-A. 화면 9축 (프론트)
존재 · IA/라우팅 · **상태(빈/로딩/에러/예외)** · 공통컴포넌트 · **마스킹** · 카피(PNG 대조) · 디자인토큰 · 반응형(1440×900·390×844) · 인터랙션.

### 3-B. 데이터 5축 (백엔드/스키마 — DB변경·마이그레이션·시드 포함 작업에 필수)
1. **스키마 정합** — `sql` ↔ `schema.ts`/ORM ↔ `ERD.md` ↔ 엔드포인트 계약 4자 동기화, 네이밍/타입 컨벤션.
2. **마이그레이션 안전** — 멱등·롤백 제시·파괴적 변경 사용자확인·Aurora `information_schema` 적용확인·온라인 DDL 제약.
3. **시드 정합** — 부트스트랩 필수 시드(관리자 계정·역할·권한매트릭스·기본 게시판·알림라우팅) 존재·재실행 안전.
4. **데이터 정합성** — 자연키/UNIQUE(업서트·정산·멱등키)·고아레코드/약한FK 가드·소프트삭제 vs 물리파기.
5. **성능** — 조회패턴 인덱스 커버·실행계획·대량작업 잠금/영향범위.

### 3-C. 보안 상시 체크(모든 실연동 라운드 최소 적용) — security 렌즈
인증/세션(보호라우트 전수·JWT HttpOnly/Secure/SameSite/만료·회전·무토큰 401) · 인가/RBAC(05 #3 + own 스코프 서버강제·IDOR·크로스테넌트 격리·직접 URL 403) · 시크릿(클라 번들·로그·응답·커밋 grep 0건, 비번·인증코드 응답 미포함) · 남용(`/auth/*` 레이트리밋·계정열거 통일).

### 3-D. 마스킹 3레이어 — privacy 렌즈
- **L1 응답**: **05 C-6 확정** 마스킹 규칙(본인 자기조회 vs 타인 × 식별정보/카드·계좌/비번·인증코드) — 표는 06 참고.
- **L2 로그·에러**: 이름·이메일·휴대폰·카드·계좌·비번·인증코드·생년월일·`provider_uid`는 로그·에러바디·스택트레이스에 평문 금지. 웹훅 `raw_payload` 마스킹.
- **L3 외부전송**: PG(toss)·NICE·GA/Meta·(향후 AI 번역/튜터) 전송은 최소화 + **국외이전이면 고지·동의 근거 확인**(개인정보보호법 §28의8). → **법무 자문 별도 안건**.
- 마스킹 알고리즘은 **단일 정본(05/C-6 기준)**을 따른다(예: 이메일 = 앞 2자 후 `***@domain`).

## 4. 심각도 기준 (`상/중/하` 단일 표기)

결함 **등급 표기**는 **상/중/하**로 통일한다(결함표·판정에 영문 등급 병기 금지). 단 `blocker`는 **'상'의 동의 개념어**로 본문·이월 원장에서 계속 쓴다.

| 심각도 | 정의(성문화) |
|---|---|
| **상**(blocker) | 05 확정 6건 위반 · 비가역 행위 무확인 · 인가/테넌트 경계 붕괴 · 데이터 손실 · 시크릿 노출 · 행복경로 차단 |
| **중**(major) | 정책·상태·계약 불일치(우회 가능) · 마스킹 L1 누락 · 회귀 위험 |
| **하**(minor) | `[추정]`·`[미확정]` 근거 · 목업 baseline 갭 · 범위 밖 개선 제안 |

- 판정 기호 `⭕`(합격)/`🔺`(조건부)/`❌`(재작업)는 **판정 결과** 표기이고, 상/중/하는 **결함 등급**이다(별개).
- blocker 강등(예: "Phase 이관 등록 시 통과")은 **원장에 사유+승인자 기재** 시에만 허용한다.

## 6. 결함표 표준 템플릿 · 네이밍

**결함표는 아래 1종 표준만 사용**한다(요약형 `심각도|결함|조치`는 재현·회귀 불가 → 금지).

```
| 결함ID | 추적키(화면ID/EP) | 검증축 | 심각도(상/중/하) | 현상(재현절차) | 기대 vs 실제 | 근거(파일:라인/SoT) | 제안 | 담당 | 상태(open/이관/closed@라운드) |
```

- **결함ID = 안정 ID**: `<영역>-<라운드>-D##`(예: `FR01-r07-D03`). 라운드 간 1:1 추적에 사용.
- **파일 네이밍**: `docs/dev-validation/<영역소문자>-r<NN>.md` — **round 번호는 영역 전역 단조증가**. version/phase는 파일명이 아니라 **헤더 메타**로. (기존 38개 파일은 소급 개명하지 않고, 신규 라운드부터 적용.)
- **재검증 종료 정의**: 직전 라운드 open 결함 전건을 동일 축으로 재점검 + **회귀 범위 = 변경파일 import 그래프 1-hop + 05 6건 상시**.

## 7. blocker 이월 원장 — `docs/dev-validation/_ledger.md`

모든 **상(blocker)** 결함은 원장에 등재해 라운드 간 추적한다. 종료조건 "blocker 0"은 원장에서 검산한다.

- 컬럼: `결함ID · 발견라운드 · 추적키 · 심각도 · 요약 · 상태(open/이관/closed) · closed라운드 · 승인자`.
- **"게이트 미통과 배포"·"오너 won't-fix"·"blocker 강등"은 원장에 사유 + 승인자 필수 기재.**

## 8. 역할 (에이전트 매핑)

- **개발**: `frontend-developer`(FR01)·`admin-developer`(AD01)·`api-developer`(API)·`lms-developer`(LMS 도메인)·`dba`(스키마). 디자인 일관성은 `uiux-lead`/`ux-designer` 표준.
- **게이트**: `qa`(동작·회귀·카피·9축)·`security-reviewer`(권한·시크릿·실연동)·`privacy-officer`(마스킹·약관·PII·비가역)·`dba`(데이터 5축).
- **조율·판정**: `plan-lead`(추적키/정책 정합)·`dev-lead`(기술 게이트)·`qa-lead`(품질 게이트 컨펌)·`ops-lead`(배포·릴리스 게이트).
- **총괄**(`chief`): 라운드 운영·최종 판정·통합 보고(취합은 총괄이 직접).

> 팀은 **글로벌 에이전트팀**(`~/.claude/agents/`)만 사용한다([docs/AGENT_TEAM.md](AGENT_TEAM.md)).
